# Requirements Document

## Introduction

Suggy Sweep — настольное приложение для Windows, предназначенное для подготовки к переустановке операционной системы. Приложение сканирует файловую систему, находит все пользовательские файлы (исключая системные и заводские), объясняет их назначение с помощью AI, и позволяет сохранить их на внешний носитель с возможностью восстановления после переустановки. Приложение построено на стеке Electron + React + Python FastAPI и использует Liquid Glass UI в тёмной теме.

---

## Glossary

- **Application**: десктопное приложение Suggy Sweep (Electron-оболочка).
- **Frontend**: React 18 + TypeScript + Vite интерфейс, запущенный внутри Electron.
- **Backend**: Python 3.11+ FastAPI-сервер, запущенный как дочерний процесс Electron на 127.0.0.1:8765.
- **Scanner**: модуль Backend, выполняющий обход файловой системы.
- **AI_Client**: модуль Backend, взаимодействующий с Lovable AI Gateway.
- **Backup_Manager**: модуль Backend, управляющий созданием и верификацией резервных копий.
- **Restore_Manager**: модуль Backend, управляющий восстановлением из резервных копий.
- **Browser_Detector**: модуль Backend, обнаруживающий установленные браузеры и их данные.
- **App_Detector**: модуль Backend, обнаруживающий поддерживаемые приложения и их данные.
- **System_Collector**: модуль Backend, собирающий системную информацию (WiFi, реестр, драйверы и т.д.).
- **Drive_Manager**: модуль Backend, обнаруживающий и предоставляющий информацию о дисках.
- **Manifest**: файл формата `.suggy` (JSON), описывающий содержимое резервной копии.
- **Liquid_Glass_UI**: стиль оформления интерфейса с эффектами стекломорфизма (backdrop-filter blur + saturate).
- **SSE**: Server-Sent Events — механизм потоковой передачи данных от Backend к Frontend.
- **WebSocket**: протокол двусторонней связи, используемый для real-time сканирования.
- **User_File**: файл, созданный или изменённый пользователем, не являющийся частью Windows или установленных программ.
- **Category**: классификационная группа файлов (Документы, Изображения, Видео, Музыка, Архивы, Код/Проекты, Другое).

## Requirements

### Requirement 1: Запуск и инициализация приложения

**User Story:** As a пользователь, I want приложение запускалось и было готово к работе без ручной настройки, so that я могу сразу начать подготовку к переустановке Windows.

#### Acceptance Criteria

1. THE Application SHALL запускать Backend как дочерний процесс на 127.0.0.1:8765 при старте Electron.
2. WHEN Backend не отвечает в течение 10 секунд после запуска, THE Application SHALL отобразить сообщение об ошибке на русском языке и предложить повторить попытку.
3. THE Application SHALL открывать главное окно с минимальным размером 1024×768 пикселей и оптимальным размером 1440×900 пикселей.
4. THE Application SHALL отображать кастомный title bar в стиле macOS с кнопками управления окном (закрыть, свернуть, развернуть) и поддержкой перетаскивания.
5. WHEN Application закрывается, THE Application SHALL завершать дочерний процесс Backend.
6. THE Application SHALL отображать весь текст интерфейса на русском языке.
7. THE Application SHALL применять тёмную тему Liquid_Glass_UI по умолчанию при каждом запуске.

### Requirement 2: Навигация и боковая панель

**User Story:** As a пользователь, I want видеть все разделы приложения в боковой панели с актуальной статистикой, so that я могу быстро переходить между функциями.

#### Acceptance Criteria

1. THE Frontend SHALL отображать боковую панель с разделами: Дашборд, Сканер, Мои файлы, Браузеры, Приложения, Бэкап, Восстановление, AI Ассистент, Системная информация, Настройки.
2. WHEN пользователь нажимает на пункт навигации, THE Frontend SHALL переключать активный экран без перезагрузки страницы.
3. THE Frontend SHALL отображать в боковой панели живую статистику: количество найденных файлов, общий размер, статус последнего бэкапа.
4. WHILE сканирование активно, THE Frontend SHALL отображать индикатор прогресса в боковой панели рядом с разделом «Сканер».
5. THE Frontend SHALL выделять активный пункт навигации визуально с использованием Liquid_Glass_UI эффектов.

### Requirement 3: Дашборд

**User Story:** As a пользователь, I want видеть приветственный экран с обзором системы и быстрыми действиями, so that я могу понять текущее состояние системы и начать нужное действие одним кликом.

#### Acceptance Criteria

1. THE Frontend SHALL отображать на Дашборде сетку из 6 карточек быстрых действий: «Начать сканирование», «Создать бэкап», «Восстановить данные», «AI Ассистент», «Браузеры», «Приложения».
2. WHEN пользователь нажимает карточку быстрого действия, THE Frontend SHALL переходить к соответствующему разделу.
3. THE Frontend SHALL отображать на Дашборде системный обзор: имя пользователя, версия Windows, архитектура, имя компьютера.
4. WHEN данные системного обзора недоступны, THE Frontend SHALL отображать заглушку с сообщением об ошибке загрузки.
5. THE Frontend SHALL анимировать переходы между экранами с использованием framer-motion.

### Requirement 4: Сканер файловой системы

**User Story:** As a пользователь, I want сканировать систему и видеть все мои файлы в реальном времени с разбивкой по категориям, so that я понимаю, что нужно сохранить перед переустановкой.

#### Acceptance Criteria

1. THE Frontend SHALL отображать перед сканированием панель настроек: выбор дисков для сканирования, переключатели категорий (Документы, Изображения, Видео, Музыка, Архивы, Код/Проекты, AppData).
2. WHEN пользователь запускает сканирование, THE Scanner SHALL передавать найденные файлы через WebSocket в реальном времени с полями: путь, имя, расширение, размер, дата изменения, категория.
3. WHILE сканирование активно, THE Frontend SHALL отображать: текущий сканируемый файл, счётчик найденных файлов, суммарный размер, прогресс-бар.
4. WHEN сканирование завершено, THE Frontend SHALL отображать сетку категорий с количеством файлов и суммарным размером в каждой.
5. WHEN пользователь выбирает категорию, THE Frontend SHALL отображать дерево файлов с возможностью раскрытия папок.
6. WHEN пользователь выбирает файл в дереве, THE Frontend SHALL отображать детали файла: путь, размер, дата изменения, категория.
7. IF Scanner не может получить доступ к файлу или директории, THEN THE Scanner SHALL пропустить этот элемент и продолжить сканирование.
8. THE Scanner SHALL исключать из результатов файлы, пути которых совпадают с настроенными исключениями из Settings.
9. WHEN WebSocket соединение разрывается во время сканирования, THE Frontend SHALL отображать сообщение об ошибке и предлагать перезапустить сканирование.

### Requirement 5: AI-объяснение файлов

**User Story:** As a пользователь, I want получать AI-объяснение для каждого файла, so that я понимаю, нужен ли мне этот файл и стоит ли его сохранять.

#### Acceptance Criteria

1. WHEN пользователь запрашивает объяснение файла, THE AI_Client SHALL отправить запрос к Lovable AI Gateway с информацией о файле (имя, путь, расширение, категория, размер).
2. THE AI_Client SHALL передавать ответ AI через SSE потоком в Frontend.
3. WHILE AI_Client получает потоковый ответ, THE Frontend SHALL отображать текст по мере поступления символов.
4. IF Lovable AI Gateway возвращает ошибку HTTP, THEN THE AI_Client SHALL вернуть сообщение об ошибке на русском языке.
5. THE AI_Client SHALL использовать системный промпт на русском языке для всех запросов.
6. THE AI_Client SHALL использовать модель google/gemini-3-flash-preview по умолчанию, если иное не задано в Settings.

### Requirement 6: AI Ассистент

**User Story:** As a пользователь, I want общаться с AI ассистентом в чате и получать рекомендации по бэкапу, so that я могу принимать обоснованные решения о сохранении данных.

#### Acceptance Criteria

1. THE Frontend SHALL отображать полноценный чат-интерфейс с историей сообщений, полем ввода и кнопкой отправки.
2. WHEN пользователь отправляет сообщение, THE AI_Client SHALL передавать ответ потоком через SSE.
3. WHILE AI_Client передаёт потоковый ответ, THE Frontend SHALL отображать индикатор набора текста и добавлять символы по мере поступления.
4. THE Frontend SHALL поддерживать анализ файла через чат: пользователь может передать путь к файлу, и AI_Client SHALL проанализировать его.
5. THE Frontend SHALL поддерживать анализ директории через чат: пользователь может передать путь к директории, и AI_Client SHALL описать её содержимое.
6. THE AI_Client SHALL формировать рекомендации по бэкапу на основе результатов сканирования по запросу пользователя.
7. IF поле ввода пустое, THEN THE Frontend SHALL блокировать отправку сообщения.

### Requirement 7: Менеджер браузеров

**User Story:** As a пользователь, I want автоматически обнаруживать установленные браузеры и экспортировать их данные, so that я не потеряю закладки, пароли и историю после переустановки.

#### Acceptance Criteria

1. THE Browser_Detector SHALL автоматически обнаруживать следующие браузеры: Google Chrome, Microsoft Edge, Mozilla Firefox, Opera, Opera GX, Brave, Vivaldi, Yandex Browser.
2. WHEN браузер обнаружен, THE Frontend SHALL отображать его имя, иконку, список профилей и доступные для экспорта данные.
3. THE Browser_Detector SHALL поддерживать экспорт следующих типов данных для каждого профиля: закладки, пароли, cookies, история, расширения, настройки.
4. WHEN пользователь запускает экспорт данных браузера, THE Browser_Detector SHALL сохранить выбранные данные в указанную директорию.
5. IF браузер запущен во время экспорта cookies или паролей, THEN THE Browser_Detector SHALL предупредить пользователя о необходимости закрыть браузер.
6. IF браузер не установлен или его профиль не найден, THEN THE Browser_Detector SHALL не отображать этот браузер в списке.
7. THE Frontend SHALL отображать статус экспорта для каждого типа данных каждого браузера.

---

### Requirement 8: Менеджер приложений

**User Story:** As a пользователь, I want автоматически обнаруживать поддерживаемые приложения и экспортировать их настройки и данные, so that я смогу быстро восстановить рабочую среду после переустановки.

#### Acceptance Criteria

1. THE App_Detector SHALL автоматически обнаруживать следующие приложения: Telegram Desktop, AyuGram, 64Gram, Kotatogram, Discord, Discord Canary, Discord PTB, Steam, VS Code, OBS Studio, qBittorrent, FileZilla, PuTTY, Notepad++.
2. THE App_Detector SHALL экспортировать для каждого обнаруженного приложения его специфичные данные: сохранения Steam, настройки VS Code (settings.json, keybindings.json, extensions list), конфигурацию OBS, настройки qBittorrent, сессии FileZilla, сессии PuTTY, настройки Notepad++.
3. THE App_Detector SHALL экспортировать Git config (~/.gitconfig) и SSH ключи (~/.ssh/).
4. THE App_Detector SHALL экспортировать конфигурацию WSL (~/.wslconfig и дистрибутивы).
5. THE App_Detector SHALL формировать список всех установленных программ с командами для переустановки через winget и/или chocolatey.
6. WHEN пользователь запрашивает скрипт переустановки, THE App_Detector SHALL генерировать PowerShell-скрипт с командами winget install для всех обнаруженных программ.
7. IF приложение не установлено, THEN THE App_Detector SHALL не отображать его в списке обнаруженных приложений.
8. THE Frontend SHALL отображать для каждого приложения: имя, версию, путь установки, список доступных для экспорта данных, статус экспорта.

### Requirement 9: Мастер резервного копирования

**User Story:** As a пользователь, I want создавать резервную копию выбранных данных на внешний носитель через пошаговый мастер, so that я могу безопасно переустановить Windows без потери данных.

#### Acceptance Criteria

1. THE Frontend SHALL реализовывать мастер бэкапа в 4 шага: (1) выбор диска назначения, (2) выбор данных для бэкапа, (3) настройки бэкапа, (4) подтверждение и запуск.
2. THE Drive_Manager SHALL предоставлять список доступных дисков с информацией: буква диска, метка, тип (съёмный/фиксированный), общий объём, свободное место.
3. WHEN пользователь выбирает диск с недостаточным свободным местом для выбранных данных, THE Frontend SHALL отображать предупреждение с указанием необходимого и доступного объёма.
4. THE Frontend SHALL на шаге 3 предоставлять настройки: формат архива (ZIP/7Z), уровень сжатия, шифрование (AES-256) с паролем, разбивка на части по заданному размеру.
5. WHEN пользователь запускает бэкап, THE Backup_Manager SHALL создавать Manifest файл (.suggy) с метаданными: дата, список файлов с хешами SHA-256, настройки бэкапа, версия приложения.
6. WHILE бэкап выполняется, THE Frontend SHALL отображать прогресс: текущий файл, скопировано/всего файлов, скопировано/всего байт, скорость, оставшееся время.
7. WHEN бэкап завершён, THE Backup_Manager SHALL верифицировать целостность каждого файла путём сравнения SHA-256 хешей.
8. WHEN верификация завершена, THE Frontend SHALL отображать итоговую статистику: количество файлов, суммарный размер, время выполнения, результат верификации.
9. IF во время бэкапа возникает ошибка записи на диск, THEN THE Backup_Manager SHALL записать ошибку в лог и продолжить бэкап оставшихся файлов.
10. WHERE шифрование включено, THE Backup_Manager SHALL шифровать архив с использованием AES-256 и заданного пользователем пароля.

### Requirement 10: Мастер восстановления

**User Story:** As a пользователь, I want восстанавливать данные из резервной копии после переустановки Windows, so that я могу вернуть свои файлы и настройки.

#### Acceptance Criteria

1. THE Frontend SHALL позволять пользователю выбрать файл Manifest (.suggy) с внешнего носителя или локального диска.
2. WHEN Manifest выбран, THE Restore_Manager SHALL прочитать и отобразить его содержимое: дата создания, список категорий, количество файлов, суммарный размер, версия приложения.
3. THE Frontend SHALL позволять пользователю выбрать отдельные файлы или категории для восстановления.
4. WHEN пользователь запускает восстановление, THE Restore_Manager SHALL восстанавливать файлы в их оригинальные пути или в указанную директорию.
5. WHILE восстановление выполняется, THE Frontend SHALL отображать прогресс: текущий файл, восстановлено/всего файлов.
6. IF файл в архиве повреждён (хеш не совпадает), THEN THE Restore_Manager SHALL пропустить файл, записать ошибку в лог и продолжить восстановление.
7. WHERE бэкап зашифрован, THE Restore_Manager SHALL запросить пароль перед началом восстановления.
8. WHEN восстановление завершено, THE Frontend SHALL отображать итоговый отчёт: восстановлено файлов, пропущено файлов, ошибки.

### Requirement 11: Сбор системной информации

**User Story:** As a пользователь, I want экспортировать системную информацию и настройки Windows, so that я смогу восстановить персонализацию и конфигурацию после переустановки.

#### Acceptance Criteria

1. THE System_Collector SHALL экспортировать пароли WiFi-сетей с использованием команды netsh wlan show profile.
2. THE System_Collector SHALL экспортировать пользовательские ветки реестра: HKCU\Software, HKCU\Environment.
3. THE System_Collector SHALL формировать список установленных драйверов с помощью driverquery.
4. THE System_Collector SHALL экспортировать SSH ключи из директории %USERPROFILE%\.ssh\.
5. THE System_Collector SHALL экспортировать содержимое Sticky Notes (plum.sqlite из AppData).
6. THE System_Collector SHALL экспортировать пользовательские шрифты из %LOCALAPPDATA%\Microsoft\Windows\Fonts\.
7. THE System_Collector SHALL формировать список принтеров с помощью Get-Printer PowerShell команды.
8. THE System_Collector SHALL экспортировать информацию об обоях рабочего стола и настройках персонализации из реестра.
9. THE System_Collector SHALL экспортировать пользовательские переменные окружения из HKCU\Environment.
10. THE System_Collector SHALL экспортировать запланированные задачи пользователя с помощью schtasks.
11. IF команда экспорта требует прав администратора и они не предоставлены, THEN THE System_Collector SHALL пропустить этот элемент и отобразить предупреждение пользователю.
12. THE Frontend SHALL отображать раздел «Системная информация» с карточками для каждого типа данных, статусом сбора и кнопкой экспорта.

### Requirement 12: Настройки приложения

**User Story:** As a пользователь, I want настраивать параметры приложения, so that я могу адаптировать его под свои нужды.

#### Acceptance Criteria

1. THE Frontend SHALL предоставлять раздел «Настройки» с группами: Интерфейс, AI, Бэкап, Сканер.
2. THE Frontend SHALL позволять выбирать тему оформления (тёмная/светлая) в группе «Интерфейс».
3. THE Frontend SHALL позволять задавать API ключ Lovable AI Gateway и выбирать модель AI в группе «AI».
4. THE Frontend SHALL позволять задавать директорию бэкапа по умолчанию, формат архива и уровень сжатия по умолчанию в группе «Бэкап».
5. THE Frontend SHALL позволять добавлять и удалять пути-исключения для сканера в группе «Сканер».
6. WHEN пользователь сохраняет настройки, THE Frontend SHALL сохранять их в локальное хранилище и применять немедленно.
7. THE Frontend SHALL хранить настройки AI ключа в зашифрованном виде с использованием Electron safeStorage API.

### Requirement 13: Liquid Glass UI и визуальное оформление

**User Story:** As a пользователь, I want видеть современный красивый интерфейс в стиле Liquid Glass, so that работа с приложением была приятной и интуитивной.

#### Acceptance Criteria

1. THE Frontend SHALL применять анимированный тёмный градиентный фон ко всему приложению.
2. THE Frontend SHALL применять эффекты стекломорфизма (backdrop-filter: blur + saturate) ко всем панелям, карточкам, кнопкам и модальным окнам.
3. THE Frontend SHALL использовать шрифт Inter (Google Fonts) для всего текста интерфейса.
4. THE Frontend SHALL использовать иконки из библиотеки @phosphor-icons/react.
5. THE Frontend SHALL использовать CSS-переменные для управления параметрами Liquid_Glass_UI эффектов.
6. THE Frontend SHALL отображать уведомления через react-hot-toast с применением Liquid_Glass_UI стиля.
7. THE Frontend SHALL анимировать все переходы между экранами и появление элементов с использованием framer-motion.
8. THE Frontend SHALL использовать Tailwind CSS 3 для базовой стилизации с кастомными CSS-расширениями для glass-эффектов.

### Requirement 14: Backend API и коммуникация

**User Story:** As a разработчик, I want Backend предоставлял стабильный API для всех операций, so that Frontend мог надёжно получать данные и управлять процессами.

#### Acceptance Criteria

1. THE Backend SHALL предоставлять REST эндпоинты для: системной информации, списка дисков, экспорта данных браузеров, экспорта данных приложений, сбора системной информации.
2. THE Backend SHALL предоставлять WebSocket эндпоинт /api/scan/ws/scan для real-time передачи результатов сканирования.
3. THE Backend SHALL предоставлять SSE эндпоинты для потоковой передачи ответов AI: /api/ai/analyze-file и /api/ai/chat.
4. THE Backend SHALL запускаться на 127.0.0.1:8765 и принимать запросы только с localhost.
5. THE Backend SHALL возвращать все ошибки в формате JSON с полями: error (строка), detail (строка), code (число).
6. WHEN Backend получает запрос на операцию, требующую длительного выполнения, THE Backend SHALL немедленно вернуть идентификатор задачи и предоставить эндпоинт для проверки статуса.
7. THE Backend SHALL логировать все ошибки в файл suggy-sweep.log в директории приложения.

### Requirement 15: Парсинг и сериализация Manifest

**User Story:** As a разработчик, I want Manifest файл корректно сериализовался и десериализовался, so that данные резервной копии всегда были целостными и восстанавливаемыми.

#### Acceptance Criteria

1. THE Backup_Manager SHALL сериализовать Manifest в формат JSON с полями: version, created_at, app_version, files (массив объектов с path, size, sha256, category), settings (объект с параметрами бэкапа).
2. WHEN Manifest файл предоставлен для восстановления, THE Restore_Manager SHALL десериализовать его в объект Manifest.
3. IF Manifest файл содержит невалидный JSON, THEN THE Restore_Manager SHALL вернуть ошибку с описанием проблемы на русском языке.
4. FOR ALL валидных объектов Manifest, сериализация затем десериализация SHALL производить эквивалентный объект (round-trip свойство).
5. THE Backup_Manager SHALL валидировать структуру Manifest перед записью: все обязательные поля должны присутствовать и иметь корректные типы.
